Build a production-ready web app called "MoveScript" using Next.js (App Router) + TypeScript.

This app is a choreography editor and viewer. It lets users create choreography projects, place people on an infinite whiteboard, manage scenes over a song timeline, animate movement between scenes, and share a viewer-only playback link.

Tech requirements:
- Next.js latest with App Router
- TypeScript
- Tailwind CSS
- Cloudflare-compatible deployment
- Firebase Authentication for:
  - Google sign-in
  - email/password sign-up and sign-in
- Supabase for:
  - PostgreSQL database
  - Storage for song files
- Clean, modular architecture
- Must be stable and actually usable, not just UI mockups
- Good error handling
- Good loading and empty states
- Responsive design for desktop first, but usable on tablet/mobile
- Persist all edits so reopening on another device shows the same state

Core concept:
- A project has exactly 1 song
- A project contains choreography scenes
- Each scene contains placed people on a whiteboard
- Users can move people, duplicate previous scenes, assign scene time, and define transitions
- There is also a viewer page that can be shared by link for playback only

Data model overview:
- User
- Project
- Song
- Scene
- PersonInstance in each scene
- Transition / Motion settings
- Share link for viewer mode

Main product rules:
1. A user can sign up, sign in, and sign out reliably.
2. A project can be created and edited by its owner.
3. When starting a new project, the user can import one or more scenes from another existing project they own.
4. Project structure is:
   Project 1 > Choreography > Scene 1, Scene 2, Scene 3 ...
5. Each project supports only one song file.
6. Each scene can contain multiple people placed on a whiteboard.
7. The whiteboard is effectively infinite.
8. The whiteboard supports zoom in, zoom out, and pan.
9. Whiteboard transform state and all choreography data must persist correctly.
10. On another device, reopening the same project must show the saved result correctly.
11. Movement of the whiteboard should only happen when:
   - the user is touching/dragging the whiteboard directly in board-move mode
   - not while dragging a person normally
12. People are created using an Add button.
13. Creation methods:
   - create a single person
   - create a grid by rows x columns
14. Each person shape can be selected from:
   - circle
   - rectangle
   - square
15. Each person can display either a name or a number inside the shape.
16. When adding a new scene, support:
   - blank scene
   - duplicate previous scene
17. Each scene has an assigned timestamp or duration related to the song timeline.
18. A person can:
   - move to a new position
   - exit the stage
   - appear in a scene
19. Exit direction options should include:
   - front
   - back
   - left
   - right
   - custom quick exit feel
20. Dragging a person sets movement to a new position.
21. Movement duration can be configured.
22. Default movement duration = half of that scene’s assigned song time.
23. Clicking a person opens editing for:
   - name or number
   - shape
   - display mode
   - appearance/entry state
   - exit state
   - motion duration
24. Viewer mode must support:
   - play
   - pause
   - scene progression playback
   - shareable link
25. Shared viewer link should be read-only.
26. Editor and viewer should be separate routes/pages.
27. Build sensible extra UX details where unspecified, but do not violate the above rules.

Project Editor:

  - viewer link/share button

- Right sidebar

  - scene timing settings

  - song progress

Whiteboard/editor behavior:
- Infinite workspace feel
- Pan and zoom
- Maintain transform smoothly
- Grid optional but subtle
- Person items rendered clearly
- Drag person to move
- Select person by click
- Multi-select is optional but nice to have
- Board-move mode toggle required
- In board-move mode, drag pans board
- In normal mode, drag person moves the person
- Use pointer events carefully to prevent accidental board movement while editing
- Persist coordinates in world space, not viewport space

People creation:
- Add person modal/panel
- Single create:
  - name or number
  - shape
- Grid create:
  - rows
  - columns
  - spacing
  - default naming/numbering strategy
  - shape
- After grid creation, place them in a sensible layout near current viewport center

Scene system:
- Scene list ordered vertically or horizontally
- Each scene has:
  - title or default name like Scene 1
  - order index
  - assigned song timing data
- Add scene options:
  - blank scene
  - duplicate previous scene
- Scene duplication should carry person placements and editable state
- Support importing scenes from another project into current project
- Imported scene becomes editable inside the current project
- Scene switching loads that scene’s layout and person states

Timing model:
- Each project has one song
- Each scene should have timing data
- Support at minimum:
  - start time
  - duration
- Display timing in seconds with friendly formatting
- Default movement duration for a moved person = scene duration / 2
- Allow overriding per person movement duration

Person state per scene:
Each person instance in a scene should support:
- stable identity across scenes when duplicated/imported logically
- x position
- y position
- visible/present or not
- display text (name or number)
- display mode
- shape
- movement duration
- entry mode
- exit mode
- optional notes

Entry/exit behavior:
- Person can appear in a scene
- Person can exit in a direction
- Viewer should animate/represent these states clearly
- Exit direction options:
  - front
  - back
  - left
  - right
  - custom
- For custom exit, allow angle/direction preset or a freeform option if simple enough

Viewer mode:
- Read-only
- Play choreography against song timing
- Pause playback
- Jump between scenes
- Show people moving according to scene data and movement durations
- Use scene timing and song timing to drive playback
- Provide shareable public link
- Public link must not allow editing
- Owner can enable/disable share link

- Show save indicator:

- All important edits must persist:

  - transitions
